# 9.3 Reference Points

_Page Type: Chapter Index | Maturity: Draft_

This section gathers the named gateway tools, adjacent control layers, and review artifacts that matter after the control-plane question is already framed. Use it to compare routing and policy options without reducing the chapter to a provider-switching feature list or hiding the identity, evidence, and portability consequences behind one product choice.

## Included Reference Subsections

- 9.3.1 [Tools And Platforms](09-03-01-tools-and-platforms.md)
- 9.3.2 [Controls And Artifacts](09-03-02-controls-and-artifacts.md)

## What This Section Helps Decide

- which gateway products, API-control layers, or identity and policy components fit the shared-control problem without hiding a larger operating or sourcing mismatch
- whether the team needs an AI-native gateway, an extended enterprise API gateway, or stronger identity and policy enforcement around direct provider access
- where openness, sovereignty, exportability, and managed-control dependence materially change the shortlist even when the routing problem seems straightforward
- when a named-tool question is really a hosting, observability, security, or exit-planning issue rather than a gateway-selection issue

## Reading Boundaries

- Use [9.3.1 Tools And Platforms](09-03-01-tools-and-platforms.md) when the team needs to compare gateway products, API mediation layers, and identity or policy systems by stack role, control posture, and lock-in profile.
- Use [9.3.2 Controls And Artifacts](09-03-02-controls-and-artifacts.md) when the real question is what review packet, policy register, exception record, or rollback evidence should exist before a gateway becomes the shared path.
- Revisit [9.1 Gateway Foundations](09-01-00-gateway-foundations.md) or [9.2 Operating Gateways And Controls](09-02-00-operating-gateways-and-controls.md) if the team still cannot name route ownership, policy scope, caller identity assumptions, exception handling, or re-review triggers before comparing references.
