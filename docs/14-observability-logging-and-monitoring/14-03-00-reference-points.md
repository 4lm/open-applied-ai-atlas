# 14.3 Reference Points

_Page Type: Chapter Index | Maturity: Review-Ready_

This section gathers the named tools and review artifacts that make observability decisions reusable. Use it when the telemetry problem is already clear and the remaining question is which platform posture, control pack, and export boundary will keep runtime evidence usable under incident, audit, and sourcing pressure.

## Included Reference Subsections

- 14.3.1 [Tools And Platforms](14-03-01-tools-and-platforms.md)
- 14.3.2 [Controls And Artifacts](14-03-02-controls-and-artifacts.md)

## What To Look For Here

- which telemetry foundations, AI-specific trace platforms, and enterprise observability suites fit the architecture without turning the evidence model into a vendor-specific dead end
- which control artifacts should exist before rollout, including schema dictionaries, retention records, alert catalogs, release-to-trace mapping, reconstruction packets, and export registers
- where openness, sovereignty, portability, privacy, and lock-in still change the answer even after the team agrees that "more logging" is not enough
- which unresolved questions should send the reader back to telemetry design, operating review, security containment, or sourcing choices instead of stretching the reference section beyond its role

## Reading Boundaries

- Start with [14.3.1 Tools And Platforms](14-03-01-tools-and-platforms.md) when the team needs to compare OpenTelemetry-aligned foundations, AI-native tracing products, and managed observability platforms without confusing them as interchangeable layers.
- Use [14.3.2 Controls And Artifacts](14-03-02-controls-and-artifacts.md) when the question is what review packet should exist before launch, renewal, or incident follow-up rather than which vendor appears strongest in a table.
- Revisit [14.1 Observability Foundations](14-01-00-observability-foundations.md) or [14.2 Operating Observability](14-02-00-operating-observability.md) if the team still cannot name required identifiers, privacy boundaries, escalation owners, or retention limits before it compares tools.
- Move to [13. Evaluation Testing And QA](../13-evaluation-testing-and-qa/13-00-00-evaluation-testing-and-qa.md), [15. Security And Abuse Resistance](../15-security-and-abuse-resistance/15-00-00-security-and-abuse-resistance.md), or [18. Build Vs Buy Vs Hybrid](../18-build-vs-buy-vs-hybrid/18-00-00-build-vs-buy-vs-hybrid.md) when the binding issue is release proof, telemetry exposure risk, or managed-platform dependence rather than observability references themselves.
