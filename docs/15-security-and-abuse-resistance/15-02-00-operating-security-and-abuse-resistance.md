# 15.2 Operating Security And Abuse Resistance

_Page Type: Chapter Index | Maturity: Review-Ready_

This section turns security design into operating review. Use it when trust boundaries are already clear and the team now needs to prove that prompt abuse, tool misuse, secrets exposure, provider egress, and incident containment still hold under real delivery pressure.

## Section Map

- 15.2.1 [Worked Security Scenarios](15-02-01-worked-security-scenarios.md)
- 15.2.2 [Patterns And Anti-Patterns](15-02-02-patterns-and-anti-patterns.md)

## What This Section Helps Decide

- which operating posture fits internal assistants, tool-using workflows, memory-bearing systems, and gateway-mediated deployments where attack surface, approval latency, and evidence retention conflict
- which recurring review shapes produce usable abuse resistance versus red-team theater, prompt-only controls, or disconnected ownership between product, platform, and security teams
- which control outputs should exist before rollout expands, including tool permission manifests, abuse-test packs, containment drills, egress exceptions, and reconstruction-ready audit paths
- which changes should reopen privacy, gateway, observability, oversight, or sourcing review, including new tools, persistent memory, provider routes, privileged actions, or cross-border handling changes
