# 2.2 Applying The Taxonomy

_Page Type: Chapter Index | Maturity: Review-Ready_

This section shows how taxonomy changes real review, governance, architecture, sourcing, operations, and publication work once a proposal or table already exists. Use it when a team has something concrete in front of it, such as a vendor shortlist, system sketch, control note, or comparison page, and needs to check whether the categories underneath it are sound.

The same check matters for chat assistants, forecasting platforms, computer-vision pipelines, classical ML services, and hybrid stacks. Taxonomy is not only for LLM tool lists.

If taxonomy is skipped here, comparison tables look deeper than they are, approvals land in the wrong lane, and operating controls or telemetry get attached to the wrong part of the system.

- how the taxonomy changes a real design, review, or sourcing decision

- which reusable comparison fields should appear by default before a table is widened, deepened, or trusted
- when a table should be widened, split, deepened, or removed instead of being preserved in a weak state
- which downstream chapter now owns the decision once the taxonomy is stable
